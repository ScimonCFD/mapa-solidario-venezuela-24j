# Revision activista del MVP

Este documento sirve para convertir feedback operativo en cambios concretos de la app.

## Pregunta central

Si esto se usara durante las primeras 72 horas de una emergencia, que habria que cambiar para que sea util y no estorbe?

## Flujo a revisar

1. Crear un reporte de necesidad.
2. Verificar o descartar el reporte.
3. Publicar solo informacion segura.
4. Registrar recursos disponibles.
5. Coordinar entrega o ruta.

## Preguntas para la persona activista

### Entrada de reportes

- Que datos faltan para entender una necesidad?
- Que datos sobran o hacen lento el registro?
- Confirmado por ahora: WhatsApp y Google Forms.
- Que preguntas exactas debe tener el Google Form?
- Como se transforma un mensaje de WhatsApp en un reporte estructurado?
- Quien deberia poder crear reportes?

### Verificacion

- Que cuenta como reporte verificado?
- Basta una llamada, una foto, un contacto local, una organizacion conocida?
- Como se seleccionan los voluntarios verificadores?
- Que datos debe dejar registrado el verificador para auditoria interna?
- Que reportes deberian descartarse?
- Que señales indican riesgo de rumor, duplicado o estafa?

### Privacidad

- Que datos nunca deben aparecer publicamente?
- Que nivel de ubicacion es seguro publicar: estado, municipio, parroquia, sector?
- Hay grupos vulnerables que requieran mayor cuidado?

### Recursos y logistica

- Que recursos hay que registrar primero: agua, comida, medicinas, transporte, refugio, voluntarios?
- Quien puede asignar un recurso a una necesidad?
- Como se confirma que la ayuda llego?

### Vista publica

- Que deberia ver una persona que quiere donar?
- Que deberia ver una organizacion?
- Que podria causar dano si se publica?

## Decisiones iniciales del MVP

- No publicar contactos internos.
- No publicar direcciones exactas.
- Tratar toda cifra como preliminar hasta verificar.
- Priorizar datos limpios sobre automatizacion.
- Mantener el flujo simple: reporte, verificacion, recurso, entrega.
